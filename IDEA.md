## 0. 기존 청첩장 처리 — 확정됨

이 레포(`Bigstargithub.github.io`)는 현재 **모바일 청첩장**을 서비스 중입니다
(순수 HTML/CSS/JS, Jekyll 아님). 블로그로 전환하며 청첩장은 루트에서 제거합니다.

**확정 방식: 브랜치로 백업 후 루트를 블로그로 교체**

1. 작업 시작 전, 현재 `main`(청첩장 상태)을 `wedding-archive` 브랜치로 백업:
   ```
   git branch wedding-archive
   git push origin wedding-archive
   ```
2. 백업 확인 후, `main`의 청첩장 파일(index.html, css/, js/, image/ 등)을 제거하고
   블로그로 재구성.

> 청첩장은 `wedding-archive` 브랜치에 그대로 보존되므로 언제든 복구 가능.
> 이 백업 브랜치 push가 확인되기 전에는 루트의 청첩장 파일을 삭제하지 말 것.

## 1. 목적

- 코딩테스트 풀이·CS 학습 내용을 **꾸준히 기록**하는 개발 블로그
- 중장기적으로 LLM/AI Agent/추론 인프라 관련 **기술 심화 글**을 쌓아
  이직·커리어의 공개 자산으로 활용
- 1차 독자: 나 자신(복습용) + 채용 담당자/동료 개발자

## 2. 기술 스택 (확정)

- **정적 사이트 생성기: Jekyll** (GitHub Pages 네이티브, 별도 빌드 파이프라인 불필요)
- **테마: Chirpy** (`jekyll-theme-chirpy`)
  - 선택 이유: 코드 블록 문법 하이라이팅 + 복사 버튼 기본 제공,
    다크/라이트 모드, 카테고리·태그, 목차(TOC), 검색, 댓글(giscus) 내장.
    한국 개발 블로그에서 검증된 조합.
- **셋업 방식: chirpy-starter 기반** (테마 전체를 fork 하지 말고 gem으로 참조)
  - 이유: 유지보수 최소화. 테마 내부를 건드리기 시작하면 글 대신 설정에
    시간을 쓰게 됨. **커스터마이징은 최소화한다.**

> 대안(minimal-mistakes 등)은 검토하지 않는다. 이미 정했으므로 비교로 시간 쓰지 말 것.

## 3. 사이트 구조

- **카테고리 (3개, 늘리지 말 것):**
  - `Algorithm` — 코딩테스트/알고리즘 풀이와 배운 점
  - `AI` — LLM·Agent·추론 인프라(vLLM 등)
  - `Backend` — DB·네트워크·OS·시스템설계·백엔드 전반
- **태그:** 사용하지 않음. Tags/Archives 탭 제거, 카테고리만으로 분류.
- **필수 페이지:** About(간단한 자기소개 + 기술 스택), 카테고리별 목록

## 4. 필수 기능 (이것만 되면 충분)

- [x] 코드 블록 문법 하이라이팅 + 복사 버튼 (Chirpy 기본)
- [x] 카테고리 분류 (태그는 사용하지 않음)
- [x] 포스트 내 목차(TOC)
- [x] 다크/라이트 모드
- [x] 한글 가독성 좋은 폰트 (예: Pretendard) — 기본 폰트가 어색하면 이 정도만 교체
- [ ] 검색 (Chirpy 기본, 켜져 있으면 유지)
- [ ] 댓글(giscus) — **나중에.** 첫 배포엔 없어도 됨.

## 5. 디자인 방향

- 깔끔하고 정보 위주. 화려한 커스터마이징 금지.
- 테마 기본 디자인을 최대한 그대로 사용. 색/폰트는 최소한만 손댄다.
- 모바일에서 코드 블록이 깨지지 않는지만 확인.

## 6. 콘텐츠 계획 (구현과 별개, 참고용)

첫 글 후보 (이미 겪은 것으로 바로 쓸 수 있음):

1. **"프로그래머스 대장균 문제로 이해한 SQL 실행 순서"**
   - `FROM → WHERE → SELECT → ORDER BY` 실행 순서
   - WHERE에서 alias를 못 쓰는 이유, ORDER BY에선 되는 이유
   - 상관 서브쿼리(correlated subquery) 개념
2. **"비트 연산으로 푸는 SQL 문제"** — GENOTYPE `&` 연산 풀이

> 배포 완료 후 첫 글은 위 1번으로. 글쓰기는 별도 작업이며, 지금은 배포까지가 목표.

## 7. 하지 않을 것 (Non-goals) — 중요

과거 패턴상 "핵심 완료 전 범위 확장"에 빠지기 쉬움. 아래는 **첫 배포 범위에서 제외:**

- 테마 내부 코드 수정 / 대규모 커스터마이징
- 커스텀 도메인 연결
- 댓글, 조회수, 애널리틱스(GA) 연동
- 다국어(영문) 지원
- 자동화 스크립트, 글 템플릿 도구 등 부가 기능

> 위 항목은 "글이 5편 이상 쌓인 뒤" 별도로 검토한다. 지금은 손대지 않는다.

## 8. Claude Code 실행 순서 (Definition of Done)

1. 청첩장 상태를 `wedding-archive` 브랜치로 백업하고 원격에 push (섹션 0)
2. 백업 push 확인 후, 루트의 청첩장 파일 제거
3. chirpy-starter 구조로 `main` 재구성 (`_config.yml`, `_posts/`, `_tabs/` 등)
4. `_config.yml` 기본값 채우기: 사이트 제목, 설명, 사용자명(Bigstargithub),
   타임존(Asia/Seoul), 언어(ko)
5. About 페이지에 자기소개 뼈대만 작성 (상세 내용은 나중에)
6. 카테고리 4개가 동작하도록 설정
7. 로컬에서 `bundle exec jekyll serve`로 정상 렌더링 확인
   (코드 블록·다크모드·모바일 확인)
8. GitHub Pages 배포 설정 확인 후 push → `https://bigstargithub.github.io` 접속 확인
9. **여기까지가 이번 작업의 끝.** 글쓰기·추가 기능은 다음 작업으로 분리.

> 각 단계는 순차적으로. 한 단계가 끝나기 전에 다음 범위로 넘어가지 말 것.
> 막히면 범위를 넓히지 말고 그 단계에서 멈춰 사용자에게 알릴 것.
